Sealed Classes

Sealed classes in Kotlin are a type of class that provides restricted class hierarchies. They are used to represent a limited set of types or states, ensuring all possible subclasses are known at compile-time.

Key Characteristics of Sealed Classes

  1. Restricted Hierarchy: All subclasses of a sealed class must be declared in the same file as the sealed class itself.

  2. Exhaustive When Expressions: When used in a when expression, all subclasses must be covered. This ensures compile-time safety.

  3. Cannot Be Instantiated Directly: Sealed classes are abstract by nature and cannot be instantiated directly.

  4. Flexible Subtypes: Subclasses can be regular classes, data classes, or objects.

sealed class Result { data class Success(val data: String) : Result() data class Error(val error: Throwable) : Result() object Loading : Result() } fun handleResult(result: Result) { when (result) { is Result.Success -> println("Success: ${result.data}") is Result.Error -> println("Error: ${result.error}") Result.Loading -> println("Loading...") } }

Use Cases for Sealed Classes

  • Modeling States: For scenarios where an object can only be in a specific number of states (e.g., loading, success, error).

  • Error Handling: Representing different types of errors in a type-safe way.

  • Domain-Specific Models: When defining closed sets of related types or responses

Behavior in when Expressions

One of the most important features of sealed classes is their integration with when expressions, which become exhaustive when used with sealed classes:

fun handleResponse(response: Result) { when (response) { is Result.Success -> println("Data: ${response.data}") is Result.Error -> println("Error: ${response.error}") Result.Loading -> println("Still loading...") // No `else` required because all cases are covered } }

Best Practices

  1. Use sealed classes when you need to restrict and control the set of possible states or types.

  2. Combine them with data classes and objects for clear, expressive models.

  3. Avoid using them for general-purpose inheritance; they are meant for specific use cases.

Enum class vs sealed class

Both enum class and sealed class are powerful constructs in Kotlin, but they serve different purposes and have distinct characteristics. Here’s a detailed comparison to help you understand their differences, use cases, and benefits.

Enum Class

  • Used to represent a fixed set of constants.

  • Each constant is a singleton instance of the enum class.

  • Typically used for simple categorizations or states with predefined values.

Sealed Class

  • Used to represent a restricted class hierarchy.

  • Allows more complex state modeling with multiple subclasses.

  • Useful when each type needs additional data or specific behavior.

Use Cases

Enum Class Use Cases

  • State Management: Represent simple states (e.g., LOADING, SUCCESS, ERROR)

enum class Status { LOADING, SUCCESS, ERROR }

  • Fixed Categories: Days of the week, directions, colors, etc

  • Switch Statements: Easy to use in when expressions without extra logic

Sealed Class Use Cases

  • Complex States: Model states with different data for each state

sealed class Result { object Loading : Result() data class Success(val data: String) : Result() data class Error(val message: String) : Result() }

  • Hierarchical Models: Model hierarchical relationships with a fixed set of subclasses.

sealed class Shape { data class Circle(val radius: Double) : Shape() data class Rectangle(val width: Double, val height: Double) : Shape() }

  • Rich Behaviors: Provide methods or logic specific to each subclass.

Matching with when

Enum

The when block automatically covers all enum values. If a value is missing, the compiler will give a warning.

enum class Status { LOADING, SUCCESS, ERROR } fun getMessage(status: Status): String { return when (status) { Status.LOADING -> "Loading..." Status.SUCCESS -> "Success!" Status.ERROR -> "Error occurred." } }

Sealed Class The when block must explicitly cover all subclasses or include an else branch.

sealed class Result { object Loading : Result() data class Success(val data: String) : Result() data class Error(val message: String) : Result() } fun getMessage(result: Result): String { return when (result) { is Result.Loading -> "Loading..." is Result.Success -> "Success: ${result.data}" is Result.Error -> "Error: ${result.message}" } }

Advantages and Limitations

Enum Class Advantages

  • Simple Syntax: Easy to define and use.

  • Type-Safe Constants: Provides predefined, type-safe constants.

  • Built-in Utility Methods: Includes values(), valueOf(), ordinal, and name. Enum Class Limitations

  • Cannot hold different data per constant.

  • Limited to static values without dynamic behavior.
